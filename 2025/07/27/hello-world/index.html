远程部署操作

Hexo 官方文档中提供了远程部署的基本方法，操作简便，但在实际使用中仍存在不少限制。最直接的方式是使用 Hexo 自带的deploy功能，一键即可将生成的静态页面部署到远程仓库：

使用 GitHub Actions 从私有仓库自动部署到 GitHub Pages 的个人主页

相比 Hexo 提供的一键部署，使用 GitHub Actions 实现自动部署的方式略显繁琐，但具备更强的灵活性和安全性。尤其适合希望将博客源码保存在私有仓库中，同时将生成的网页托管在公开的 GitHub Pages 上的场景。

具体来说，该方法遵循如下思路：

  1. 博客源码存储在一个私有仓库中（例如 myblog），日常编辑和版本控制都在此仓库进行。
  2. 通过 GitHub Actions 自动执行构建任务：监听该私有仓库的 main 分支，一旦有更新，就自动将更新内容推送到个人主页仓库的 main 分支。
  3. 将生成的 public/ 文件夹内容推送到个人主页仓库（username.github.io），这一仓库是公开的，只包含网页内容而不含源码。

这样的部署方式可以保证源码不被公开，同时又能将网站部署到 GitHub Pages 上，兼顾了隐私和展示的需求。

设置 Personal Access Token (PAT) 和 Secrets

为了让 GitHub Actions 有权限将构建好的静态网页推送到 username.github.io 这个公开仓库中，你需要创建一个具有写入权限的 Personal Access Token（PAT），并将其作为 Secret 存储在私有仓库中供 Actions 使用。

常见问题

  1. 部署路径错误

    • Hexo 配置中的 root 参数必须与部署路径一致。
    • 如果你想把博客部署到个人主页的根目录，root 应该设置为 '/'，生成的文件直接放在主页仓库根目录。
    • 如果把生成的内容放到了子目录（如 myblog/），而 root 仍是 '/'，访问时会找不到 index.html，导致 404。
    • 反之，如果你配置了 root: /myblog/，但是把内容直接放到根目录，访问也会失败。

  2. 确保 DEPLOY_TOKEN 权限

    • Personal Access Token 必须有访问私有仓库代码的权限，且有写权限推送到主页仓库。
    • 在 Actions 设置里，Token 还需允许访问对应的两个仓库。

  3. GitHub Pages 设置

    • 确保主页仓库的 GitHub Pages 设置指向 main 分支（你也可以自定义分支）的根目录（/root），否则即使部署成功，也可能找不到正确入口。

  4. 浏览器缓存

    • 部署完成后访问 404，有时是缓存问题，建议强制刷新或清除缓存后重试。

总体来说，你需要保持 Hexo 的 root 配置和 GitHub Actions 部署路径一致，且部署到主页仓库根目录。
